# Agent Presets Rewrite — Design Spec

## Goal

Replace the current agent presets implementation (~7000 lines, massive duplication, mixed concerns) with a clean architecture where:
1. Presets are **pure parsers** — they extract structured data from hook JSON, nothing else.
2. A shared **orchestrator** handles all side effects: bash snapshots, repo discovery, checkpoint execution.
3. Each hook invocation produces a **typed enum** describing the logical operation (pre/post file edit, pre/post bash call).

This is Part 1 of a two-part effort. Part 2 (future, not in scope) will use hook event data as a telemetry event log replacing messages/CAS. This spec builds the clean foundation for that.

## Non-Goals

- Changing the external CLI interface (`git-ai checkpoint <agent>`)
- Changing working log format, authorship note format, or rewrite tracking
- Changing the `PreparedCheckpointManifest` machinery in `checkpoint.rs`
- Replacing transcript reading logic (functions move but don't change)
- Adding new hook types or event coverage (Part 2)

## Core Types

### PresetContext

Common fields present for every hook event. Generated partially by the orchestrator (trace_id), partially by the preset (everything else).

```rust
pub struct PresetContext {
    pub agent_id: AgentId,
    pub session_id: String,
    pub trace_id: String,           // t_ + 14 hex chars, passed in by orchestrator
    pub cwd: PathBuf,               // working directory from hook data
    pub metadata: HashMap<String, String>,  // agent-specific kv (e.g., transcript_path)
}
```

### ParsedHookEvent

The sole output type from presets. An enum with typed variant structs — each variant carries exactly the fields relevant to that operation.

```rust
pub enum ParsedHookEvent {
    PreFileEdit(PreFileEdit),
    PostFileEdit(PostFileEdit),
    PreBashCall(PreBashCall),
    PostBashCall(PostBashCall),
}

// Note: CheckpointKind::AiTab is a special case used by AiTabPreset.
// The orchestrator maps PostFileEdit from AiTabPreset to CheckpointKind::AiTab
// rather than CheckpointKind::AiAgent. This is handled via the agent_id.tool field
// ("ai_tab") which the orchestrator checks when determining the checkpoint kind.

pub struct PreFileEdit {
    pub context: PresetContext,
    pub file_paths: Vec<PathBuf>,                      // absolute paths
    /// Explicit pre-edit file content, keyed by absolute path.
    /// When provided, the checkpoint uses this as the "before" state rather than
    /// reading from disk. Use cases:
    /// - create_file: empty string signals file doesn't exist yet
    /// - Race avoidance: hook captures content before concurrent edits can interfere
    /// When None, the checkpoint reads current disk content as the baseline.
    pub dirty_files: Option<HashMap<PathBuf, String>>,
}

pub struct PostFileEdit {
    pub context: PresetContext,
    pub file_paths: Vec<PathBuf>,                      // absolute paths
    /// Explicit file content at checkpoint time, keyed by absolute path.
    /// When provided, the checkpoint uses this as the "current" state for diffing
    /// against the stored baseline, rather than reading from disk.
    /// Used by agents that send file content inline in hook data (e.g., GithubCopilot,
    /// Pi, AgentV1, AiTab, Firebender).
    /// When None, the checkpoint reads current disk content.
    pub dirty_files: Option<HashMap<PathBuf, String>>,
    pub transcript_source: Option<TranscriptSource>,
}

pub struct PreBashCall {
    pub context: PresetContext,
    pub tool_use_id: String,
    pub strategy: BashPreHookStrategy, // EmitHumanCheckpoint or SnapshotOnly
}

pub struct PostBashCall {
    pub context: PresetContext,
    pub tool_use_id: String,
    pub transcript_source: Option<TranscriptSource>,
}
```

### AgentPreset Trait


## Orchestrator

The orchestrator sits between preset parsing and the existing checkpoint machinery. It handles all side effects that previously lived inside presets.

### Entry Point

```rust
pub fn execute_preset_checkpoint(
    preset_name: &str,
    hook_input: &str,
) -> Result<Vec<CheckpointResult>, GitAiError> {
    let trace_id = generate_trace_id();  // t_ + 14 random hex
    let preset = resolve_preset(preset_name)?;
    let events = preset.parse(hook_input, &trace_id)?;

    events.into_iter()
        .map(|event| execute_event(event, &trace_id))
        .collect()
}
```

### Event Execution

```rust
fn execute_event(event: ParsedHookEvent, trace_id: &str) -> Result<CheckpointResult, GitAiError> {
    match event {
        ParsedHookEvent::PreFileEdit(e) => {
            // 1. Resolve repo from file_paths (find_repository_for_file on first path)
            //    cwd is NOT used for repo discovery — files may be outside cwd
            // 2. Return CheckpointResult with:
            //    - checkpoint_kind: Human
            //    - path_role: WillEdit
            //    - file_paths from event
            //    - dirty_files if provided (explicit pre-edit content; e.g., empty string
            //      for create_file to mark file as not-yet-existing, or current content
            //      captured by the hook to avoid races with concurrent tool calls)
        }
        ParsedHookEvent::PostFileEdit(e) => {
            // 1. Resolve repo from file_paths (find_repository_for_file on first path)
            //    cwd is NOT used for repo discovery — files may be outside cwd
            // 2. Return CheckpointResult with:
            //    - checkpoint_kind: AiAgent (or AiTab if agent_id.tool == "ai_tab")
            //    - path_role: Edited
            //    - file_paths from event
            //    - dirty_files passed through (used as baseline for diffing)
            //    - transcript_source passed through
        }
        ParsedHookEvent::PreBashCall(e) => {
            // 1. Resolve repo from cwd (bash calls don't have file paths yet;
            //    cwd is the best available signal for where the command runs)
            // 2. Call prepare_agent_bash_pre_hook (takes snapshot)
            // 3. Return CheckpointResult with:
            //    - checkpoint_kind: Human
            //    - captured_checkpoint_id from snapshot
        }
        ParsedHookEvent::PostBashCall(e) => {
            // 1. Resolve repo from cwd (same as pre — bash operates in cwd)
            // 2. Call handle_bash_tool to diff snapshots
            // 3. Discover changed file paths from diff
            // 4. Return CheckpointResult with:
            //    - checkpoint_kind: AiAgent
            //    - file_paths: discovered from diff (these ARE absolute)
            //    - captured_checkpoint_id if present
        }
    }
}
```

## Parse Helpers Module

A shared module eliminating the ~50 duplicated JSON extraction blocks across presets.

```rust
pub mod parse {
    pub fn required_str(data: &Value, key: &str) -> Result<&str, GitAiError>;
    pub fn optional_str<'a>(data: &'a Value, key: &str) -> Option<&'a str>;
    pub fn str_or<'a>(data: &'a Value, key: &str, fallback: impl FnOnce() -> Result<String, GitAiError>) -> Result<String, GitAiError>;
    pub fn str_or_default<'a>(data: &'a Value, key: &str, default: &'a str) -> &'a str;
    pub fn required_file_stem(data: &Value, path_key: &str) -> Result<String, GitAiError>;
    pub fn file_paths_from_tool_input(data: &Value, cwd: &str) -> Vec<PathBuf>;
    pub fn resolve_absolute(path: &str, cwd: &str) -> PathBuf;
}
```

All path resolution happens here — relative paths get resolved against `cwd` to produce absolute `PathBuf` values.

## Transcript Readers Module

All `transcript_and_model_from_*` functions move into a dedicated module. They are called by the orchestrator when resolving a `TranscriptSource::Path`, not during preset parsing.

```rust
pub mod transcript_readers {
    pub fn read_transcript(source: &TranscriptSource) -> Result<(AiTranscript, Option<String>), GitAiError> {
        match source {
            TranscriptSource::Path { path, format } => match format {
                TranscriptFormat::ClaudeJsonl => read_claude_jsonl(path),
                TranscriptFormat::GeminiJson => read_gemini_json(path),
                TranscriptFormat::WindsurfJsonl => read_windsurf_jsonl(path),
                // ... etc
            },
            TranscriptSource::Inline(transcript) => Ok((transcript.clone(), None)),
        }
    }
}
```

The individual reader functions are unchanged — they just live in a new home and are dispatched by format enum instead of being called directly by each preset.

## Daemon Integration (First-Class Path)

The daemon is the primary execution path. The sync/wrapper mode (where the `git` wrapper runs checkpoints inline without the daemon) is legacy and will be deprecated. The architecture reflects this:

### Execution Model

```
┌─────────────────────────────────────────────────────────┐
│  Checkpoint Process (git-ai checkpoint <agent>)            │
│                                                         │
│  1. Generate trace_id                                   │
│  2. Run preset.parse(hook_input, trace_id)              │
│  3. Run orchestrator.execute_event(parsed_event)        │
│     → Produces CheckpointResult                         │
│  4. Dispatch to daemon:                                 │
│     a. If captured_checkpoint_id present → Captured req │
│     b. Else try prepare_captured_checkpoint → Captured  │
│     c. Fallback: Live request with CheckpointResult     │
│                                                         │
└─────────────────────┬───────────────────────────────────┘
                      │ Unix socket
┌─────────────────────▼───────────────────────────────────┐
│  Daemon Process                                         │
│                                                         │
│  • Receives CheckpointResult (Live) or capture_id       │
│  • Never calls presets — never needs to                 │
│  • Family sequencer orders per-repo                     │
│  • Applies checkpoint via checkpoint::run()             │
│  • Updates watermarks                                   │
│  • Cleans up captured checkpoint files                  │
│                                                         │
│  Also constructs CheckpointResult directly for:         │
│  • Commit-inside-bash replay (bash_tool context)        │
│  • Daemon-internal human replay                         │
└─────────────────────────────────────────────────────────┘
```

### Key Design Decisions

1. **Preset parsing + orchestration happen in the checkpoint process** — the daemon receives fully-resolved `CheckpointResult`, never raw hook JSON.
2. **`CheckpointResult` is serializable** (Serialize + Deserialize) — it travels over the Unix socket as part of `LiveCheckpointRunRequest`.
3. **Captured checkpoints remain unchanged** — `prepare_captured_checkpoint` consumes `CheckpointResult` (replacing `AgentRunResult`) to write the manifest to disk. The daemon loads and executes captures without needing the result type.
4. **Daemon-internal construction** — the daemon's `build_human_replay_agent_result` and bash-in-flight logic construct `CheckpointResult` directly (they're not preset-driven).
5. **The fallback local path** (`checkpoint::run()` without daemon, used in sync/wrapper mode) also consumes `CheckpointResult` — same type regardless of execution path.

### Control API Changes


## Preset Implementation Pattern

Each preset becomes a pure parser. Example (Claude):

```rust
pub struct ClaudePreset;

impl AgentPreset for ClaudePreset {
    fn parse(&self, hook_input: &str, trace_id: &str) -> Result<Vec<ParsedHookEvent>, GitAiError> {
        let data: Value = serde_json::from_str(hook_input)
            .map_err(|e| GitAiError::PresetError(format!("Invalid JSON: {}", e)))?;

        // Validation: skip if this is actually a copilot/cursor payload
        if Self::is_vscode_copilot_hook_payload(&data) || Self::is_cursor_hook_payload(&data) {
            return Err(GitAiError::PresetError("Wrong preset for this payload".into()));
        }

        let cwd = parse::required_str(&data, "cwd")?;
        let transcript_path = parse::required_str(&data, "transcript_path")?;
        let session_id = parse::str_or(&data, "session_id",
            || parse::required_file_stem(&data, "transcript_path"))?;
        let tool_name = parse::optional_str(&data, "tool_name")
            .or_else(|| parse::optional_str(&data, "toolName"));
        let hook_event = parse::optional_str(&data, "hook_event_name")
            .or_else(|| parse::optional_str(&data, "hookEventName"));
        let tool_use_id = parse::str_or_default(&data, "tool_use_id", "bash");

        let is_bash = tool_name
            .map(|n| classify_tool(Agent::Claude, n) == ToolClass::Bash)
            .unwrap_or(false);

        let context = PresetContext {
            agent_id: AgentId {
                tool: "claude".to_string(),
                id: session_id.clone(),
                model: "unknown".to_string(),
            },
            session_id,
            trace_id: trace_id.to_string(),
            cwd: PathBuf::from(cwd),
            metadata: HashMap::from([("transcript_path".into(), transcript_path.to_string())]),
        };

        let transcript_source = Some(TranscriptSource::Path {
            path: PathBuf::from(transcript_path),
            format: TranscriptFormat::ClaudeJsonl,
        });

        let event = match (hook_event, is_bash) {
            (Some("PreToolUse"), true) => ParsedHookEvent::PreBashCall(PreBashCall {
                context,
                tool_use_id: tool_use_id.to_string(),
            }),
            (Some("PreToolUse"), false) => ParsedHookEvent::PreFileEdit(PreFileEdit {
                context,
                file_paths: parse::file_paths_from_tool_input(&data, cwd),
                dirty_files: None,
            }),
            (_, true) => ParsedHookEvent::PostBashCall(PostBashCall {
                context,
                tool_use_id: tool_use_id.to_string(),
                transcript_source,
            }),
            (_, false) => ParsedHookEvent::PostFileEdit(PostFileEdit {
                context,
                file_paths: parse::file_paths_from_tool_input(&data, cwd),
                dirty_files: None, // Claude doesn't provide pre-edit content via hook
                transcript_source,
            }),
        };

        Ok(vec![event])
    }
}
```

## Integration with Sessions/Trace IDs

The trace_id generated by the orchestrator:
- Is passed into `preset.parse()` for logging during parsing
- Lives in `PresetContext` on every event
- Flows into `CheckpointResult`
- Gets written to the working log entry
- Appears in the final authorship note as the trace component of the composite author_id (`s_<session>::t_<trace>`)

The session_id from `PresetContext`:
- Is extracted by the preset from hook data (agent-specific field names)
- Used by the orchestrator for bash snapshot correlation
- Flows into the session record in the authorship note

## CheckpointResult → Existing Checkpoint Machinery

`CheckpointResult` is designed as a direct replacement for `AgentRunResult`. The mapping:

| AgentRunResult field | CheckpointResult equivalent |
|---|---|
| `agent_id` | `agent_id` |
| `checkpoint_kind` | `checkpoint_kind` |
| `repo_working_dir: Option<String>` | `repo_working_dir: PathBuf` (always present, resolved by orchestrator via `find_repository_for_file` on file paths, or from cwd for bash calls) |
| `edited_filepaths` / `will_edit_filepaths` | `file_paths` + `path_role` (unified) |
| `dirty_files: Option<HashMap<String, String>>` | `dirty_files: Option<HashMap<PathBuf, String>>` (on both PreFileEdit and PostFileEdit; flows through to CheckpointResult) |
| `transcript: Option<AiTranscript>` | `transcript_source: Option<TranscriptSource>` (lazy) |
| `agent_metadata` | `metadata` |
| `captured_checkpoint_id` | `captured_checkpoint_id` |

The `explicit_capture_target_paths()` function in `checkpoint.rs` currently dispatches on `CheckpointKind` to pick `will_edit_filepaths` vs `edited_filepaths`. With `CheckpointResult`, this logic simplifies to just using `file_paths` + `path_role` directly — the orchestrator has already resolved this.

## Testing Strategy

- **Unit tests per preset**: JSON input → assert `ParsedHookEvent` output. No git repos needed. Fast.
- **Orchestrator integration tests**: Feed `ParsedHookEvent` variants → verify `CheckpointResult` has correct kind, paths, repo.
- **End-to-end**: Existing `TestRepo`-based tests continue working (they invoke via CLI, which flows through the new dispatch).
- **Transcript reader tests**: Existing tests for parsing logic move alongside the functions.
